# Technical Decisions

Every major technical choice in DataPilot is documented here with the reasoning behind it.

## LLM Provider: Groq (Llama 3.3 70B) as primary, Google Gemini as fallback

**Decision:** Use Groq's free tier with Llama 3.3 70B as the primary LLM, with automatic fallback to Google Gemini's free tier.

**Why:** Zero-cost operation is a hard requirement. Groq offers the fastest inference for open-source models (>500 tokens/sec). Llama 3.3 70B matches GPT-4 on most benchmarks. Multi-provider fallback ensures resilience — if one provider has downtime or rate limits, the system automatically switches.

**Alternatives considered:**
- OpenAI GPT-4o: Best quality but costs ~$5-15/1000 queries. Rejected for cost.
- Ollama (local): No cost but requires GPU hardware the user may not have. Kept as optional third fallback.
- Claude API: Excellent quality but no free tier for API access.

## Vector Database: ChromaDB (local)

**Decision:** Use ChromaDB with local persistence instead of pgvector or Pinecone.

**Why:** ChromaDB requires zero infrastructure — no database server, no cloud account, no API keys. For a portfolio project that must be one-command setup, this eliminates an entire dependency. ChromaDB's default embedding model works well for document search at this scale.

**Alternatives considered:**
- pgvector via Supabase: Better for production but requires a Supabase account or local PostgreSQL. Adds setup friction.
- Pinecone: Cloud-only, requires API key, signals "tutorial project" in 2026 since it's the default in every LangChain tutorial.
- Weaviate: Powerful but over-engineered for this use case.

## Database: SQLite for demo, PostgreSQL-ready

**Decision:** Use SQLite for the demo database with SQLAlchemy so switching to PostgreSQL requires only changing the connection string.

**Why:** SQLite requires zero setup — the database file is generated by a seed script. This means `git clone && python seed_database.py && python -m backend.main` works immediately. The SQLAlchemy abstraction layer means production deployment with PostgreSQL needs only an env var change.

## Agent Orchestration: LangGraph (not AutoGen, not CrewAI)

**Decision:** Use LangGraph for multi-agent orchestration with a state graph pattern.

**Why:** LangGraph is the production consensus winner in 2026. It provides:
- Explicit state machines (vs. implicit in AutoGen/CrewAI)
- Built-in checkpointing for recovery
- Conditional routing based on agent results
- Async-first design

**Alternatives considered:**
- AutoGen: In maintenance mode as of late 2025. Not recommended for new projects.
- CrewAI: Good for demos but lacks the fine-grained control needed for production routing logic.
- Custom orchestration: More flexible but reinvents the wheel. LangGraph is battle-tested.

## MCP Framework: FastMCP

**Decision:** Use FastMCP (Python) for all MCP server implementations.

**Why:** FastMCP is the simplest way to create MCP servers — a few decorators and you're done. It's on ThoughtWorks' Technology Radar. The alternative (raw MCP SDK) requires significantly more boilerplate for the same result.

## Chunk Size: 512 tokens with 50-token overlap

**Decision:** Document chunks are 512 tokens with 50-token overlap for the Document Agent.

**Why:** 512 tokens balances context completeness against retrieval precision. Smaller chunks (128-256) lose context; larger chunks (1024+) dilute relevance in semantic search. The 50-token overlap (~10%) prevents information loss at chunk boundaries. These values were chosen based on published benchmarks from the RAG community, not arbitrary selection.

## SQL Safety: Allowlist approach

**Decision:** Only allow SELECT and WITH (CTE) queries. Explicitly block DROP, DELETE, UPDATE, INSERT, ALTER, TRUNCATE, EXEC.

**Why:** The SQL Agent converts natural language to SQL — there's an inherent injection risk if the LLM generates a destructive query. The allowlist approach (only SELECT/WITH) is safer than a blocklist. Even if an LLM hallucinates a DROP TABLE inside a SELECT, the keyword blocklist catches it as a second defense layer.

## Frontend: Next.js + Tailwind (not Streamlit)

**Decision:** Build a proper Next.js frontend instead of Streamlit.

**Why:** Streamlit signals "prototype/demo" to hiring managers. A Next.js frontend with WebSocket streaming and an agent trace panel signals "production-ready." The additional effort is ~1 week but dramatically changes the perception. Tailwind CSS enables rapid iteration without a component library dependency.

## Observability: Structured logging (defer Langfuse to Phase 2)

**Decision:** Start with structured Python logging. Add Langfuse integration in Phase 2.

**Why:** Langfuse self-hosting requires Docker Compose with PostgreSQL — adding complexity to initial setup. Structured logging provides 80% of the value (debugging, performance tracking) with 0% additional infrastructure. Langfuse integration is designed to be a drop-in addition once the core system is stable.

## Chart Generation: matplotlib (server-side)

**Decision:** Generate charts server-side with matplotlib, return as base64 PNG.

**Why:** Server-side rendering means the chart works regardless of client capabilities (WebSocket, REST, MCP). Base64 encoding allows embedding in any response format. Alternative approaches (client-side Chart.js) would require the frontend to understand data schemas, coupling the frontend to analytics logic.
